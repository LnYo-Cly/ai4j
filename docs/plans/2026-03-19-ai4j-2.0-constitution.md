# ai4j 2.0 宪法 / 总纲设计文档

- 版本：v0.2-draft
- 日期：2026-03-19
- 状态：Draft
- 适用范围：`ai4j 2.0` 全部模块、API、运行时、工作流、Coding Agent、CLI/TUI、Spring Boot 集成、文档与测试体系

## 1. 文档目的

本文件用于定义 `ai4j 2.0` 的最高级设计约束，包括：

- 项目定位与愿景
- 核心架构原则
- 模块边界
- 兼容策略
- 演进方向
- TASK 总表

后续实现、测试、文档、CLI/TUI、工作流平台化设计均以本文件为准。

## 2. 项目定位

`ai4j 2.0` 的定位不是单一聊天 SDK，而是：

- 统一模型调用 SDK
- 通用 Agent Framework
- Workflow Runtime
- Coding Agent 基础设施
- Spring Boot / CLI / TUI / 工作流平台后端能力层

## 3. 愿景

在保持统一、简单、易用的 AI 调用体验前提下，构建一个可长期演进的 Java AI 基础设施平台，使用户既可以快速调用模型，也可以构建通用智能体、工作流系统、Coding Agent 和平台化 AI 应用。

## 4. 最终预期效果

### 4.1 对普通用户

- 继续以低心智负担接入不同 AI 提供商
- 继续使用统一输入与输出抽象
- 继续方便地完成 `chat`、`embedding`、`image`、`audio`、`multimodal`、`realtime`
- 继续支持简单工具调用与自动 tool-loop

### 4.2 对进阶用户

- 可构建通用 Agent
- 可使用 `ReAct`、自定义 Agent、`SubAgent`、`Agent Team`、`Trace`、`Workflow`
- 可复用统一运行时，而无需手工拼接底层能力

### 4.3 对平台开发者

- 可将 `ai4j` 作为工作流与 Agent 平台后端
- 可对接 FlowGram 等前端工作流编排组件
- 可逐步走向类似 Coze / Dify 的平台化后端能力

### 4.4 对 Coding Agent 用户

- 可基于 `ai4j-coding` 构建面向代码场景的 Agent
- 支持 `workspace`、`shell`、`fs`、`git`、`patch`、`browser`、`approval`、`sandbox`
- 可演进到类 Claude Code / Codex / OpenCode / pi-sdk 的能力体系

## 5. 设计哲学

### 5.1 对用户统一，对内部吸收差异

`ai4j 2.0` 继续对用户提供统一的输入与输出抽象，以降低不同 AI 提供商之间的接入成本与学习成本。

不同 AI 提供商之间的协议、能力与行为差异，由 `ai4j` 内部负责吸收与管理。统一不等于忽略差异，框架内部必须通过能力矩阵、协议分层与统一运行时机制显式处理差异。

### 5.2 简单优先，但不牺牲上限

`ai4j 2.0` 必须继续保持简单易用。简单 API 不能成为高级能力的架构瓶颈。

### 5.3 运行时统一

Tool Loop、Agent Loop、Workflow 执行、Trace、Checkpoint、Session 等运行时能力必须统一收口，不得让多个 Provider 各自控制完整运行时语义。

### 5.4 平台能力前置设计

Workflow、Trace、Checkpoint、Coding Agent、Approval、Sandbox、Replay 等能力不得作为后补丁设计，必须在 2.0 初期纳入整体架构。

### 5.5 开源兼容优先

`ai4j 2.0` 不采用“推倒重来”策略。任何架构升级都必须考虑现有开源用户的迁移成本。

## 6. 核心原则

### 原则 1：统一输入输出原则

`ai4j 2.0` 延续统一输入与输出抽象的核心设计，不因 Provider 增加或能力增加而放弃这一价值。

### 原则 2：能力差异显式管理原则

框架内部必须显式管理不同 Provider 的能力差异，至少包括：

- capability matrix
- protocol family
- behavior notes

### 原则 3：Provider 只负责适配原则

Provider 层仅负责：

- 协议适配
- 参数转换
- 事件转换
- 请求发送
- 响应解析
- 能力声明

Provider 层不得承担：

- Agent Loop 总控
- Workflow 编排
- Coding Agent 执行策略
- 全局 Session / Memory 编排

### 原则 4：统一 Loop Engine 原则

Tool Loop、Agent Loop、Workflow Node 执行、Coding Agent 调度，必须建立统一运行时语义。简单 API 允许提供自动 loop 体验，但内部仍必须复用统一内核。

### 原则 5：Workflow 一等公民原则

Workflow 是 `ai4j 2.0` 的核心能力之一，而不是边缘增强，其运行时设计必须与 Agent 一并规划。

### 原则 6：Flow UI 解耦原则

FlowGram 或任何工作流前端组件只能作为前端层或适配层，后端必须拥有自己的 Canonical Workflow Schema。

### 原则 7：Coding Agent 产品层原则

Coding Agent 不是通用 Agent 的简单别名，而是建立在通用 Agent 内核上的高级产品层。

### 原则 8：兼容演进原则

保留简单接口、保留统一体验、保留迁移路径，不允许对现有用户进行断崖式破坏升级。

### 原则 9：文档与测试同步原则

架构升级、模块升级、能力升级，必须同步更新文档与测试，不得长期脱节。

### 原则 10：可观测性优先原则

Trace、Event、Usage、Replay、Eval、Debug 设计必须优先进入主架构。

## 8. 模块职责

### 8.1 ai4j-core

- 定位：基础契约与底座模块
- 包含：
  - config 基础抽象
  - 通用异常体系
  - capability 声明
  - 通用 event 类型
  - trace 基础类型
  - tool 基础接口
  - session/context 基础对象
  - SPI
- 不包含：
  - 具体 Provider DTO
  - 具体 Chat / Image / Audio / Embedding 实现
  - Agent 业务逻辑
  - Coding 业务逻辑

### 8.2 ai4j-model

- 定位：统一模型能力模块
- 包含：
  - `chat`
  - `responses`
  - `embedding`
  - `image`
  - `audio`
  - `realtime`
  - `multimodal`
  - provider 适配
  - facade
  - 简单 tool-loop 入口
- 不包含：
  - 通用 Agent Runtime 总控
  - Workflow Graph Runtime
  - Coding Workspace / Policy / Sandbox

### 8.3 ai4j-agent

- 定位：通用 Agent Runtime 与编排模块
- 包含：
  - runtime
  - react
  - custom agent
  - subagent
  - team
  - workflow
  - memory
  - middleware
  - trace
  - checkpoint/resume
- 不包含：
  - provider HTTP 实现
  - coding 专属本地执行能力
  - CLI/TUI UI 逻辑

### 8.4 ai4j-coding

- 定位：Coding Agent 产品层
- 包含：
  - coding session
  - workspace
  - `shell / fs / git / patch / browser`
  - approval policy
  - sandbox policy
  - context compaction
  - coding strategy / planner / executor
- 不包含：
  - 通用 provider 适配
  - Spring Boot 自动装配
  - 通用模型基础服务实现

### 8.5 ai4j

- 定位：统一入口、兼容入口、默认依赖组合
- 包含：
  - facade
  - compat
  - quickstart
  - 默认组合入口
- 不包含：
  - 核心内核实现继续下沉堆积

### 8.6 ai4j-spring-boot-starter

- 定位：Spring Boot 集成层
- 目标：通过自动装配降低 Spring Boot 场景接入成本

### 8.7 ai4j-bom

- 定位：依赖版本对齐层

### 8.8 ai4j-cli

- 定位：CLI/TUI 本地交互层
- 状态建议：首发可进入版本，但建议标记为 `experimental` 或 `beta`

## 9. 模块依赖规则

```text
ai4j-core
   ↑
ai4j-model
   ↑
ai4j-agent
   ↑
ai4j-coding
   ↑
ai4j-cli

ai4j -> ai4j-model + ai4j-agent
starter -> ai4j
bom -> dependencyManagement only
```

### 约束

- `ai4j-core` 不得依赖任何上层模块
- `ai4j-model` 不得反向依赖 `ai4j-agent`
- `ai4j-agent` 不得反向依赖 `ai4j-coding`
- `ai4j` 不得成为新的核心实现堆积地
- `ai4j-cli` 只消费 SDK，不定义 SDK 内核

## 10. Workflow 设计结论

- `workflow` 在 `2.0` 首发阶段不单独拆 Maven 模块
- `workflow` 归属 `ai4j-agent`
- 代码结构必须按未来可独立拆分来设计
- 原因：当前 workflow 与 agent runtime、trace、memory、checkpoint 耦合较强，现阶段强行物理拆分只会增加复杂度

## 11. OpenAI 风格统一抽象策略

- `ai4j 2.0` 继续以 OpenAI 风格输入输出作为主要统一抽象
- 对用户：继续统一调用方式
- 对内部：必须区分
  - provider identity
  - capability matrix
  - protocol family
  - behavior difference

统一输入输出不代表所有 Provider 拥有完全相同能力，也不代表运行时语义完全等价。

## 12. OpenAI 与 OpenAI-compatible 设计结论

- 对用户层：保留当前体验，用户可继续使用 `OPENAI + baseUrl` 接入 OpenAI-compatible 平台
- 对内部层：不得将官方 OpenAI 与所有 OpenAI-compatible 平台视为完全等价对象
- 原因：
  - 支持能力不同
  - protocol family 覆盖范围不同
  - 行为语义不同
- 结论：对用户可统一，对内部必须分辨

## 13. Anthropic 设计结论

- Anthropic 应在 `2.0` 中作为原生一等 Provider 接入
- 不应只作为 “OpenAI-compatible baseUrl” 存在
- 应支持原生 `Messages` / 原生能力映射
- 同时允许兼容模式作为补充路径

## 14. Coding Agent 设计结论

### 14.1 对标目标

- Claude Code
- Codex / Codex CLI
- OpenCode
- pi-sdk

### 14.2 最小必备能力

- workspace
- shell
- file read/write
- git
- patch
- approval
- sandbox
- context compaction
- trace

### 14.3 非首阶段强制项

- 完整 browser/devtools 深度能力
- 企业级远程沙箱执行环境
- 全部 TUI 模式完全稳定

## 15. FlowGram 与平台化结论

- 目标：使 `ai4j` 可以作为 FlowGram / 自定义工作流前端的后端内核
- 原则：
  - FlowGram 是前端/适配层
  - `ai4j` 必须拥有自己的后端 Canonical Workflow Schema
  - 通过 adapter 对接 FlowGram，而不是反过来绑定后端结构

## 16. 文档原则

- 每个模块必须有 Quick Start
- 每个核心能力必须有 Demo
- 每个高级特性必须有边界说明
- 每个对外公开类/核心类必须有用途说明
- 文档结构必须按产品面组织：
  - model
  - agent
  - workflow
  - coding
  - starter
  - cli

## 17. 测试原则

- 测试必须分层：
  - unit
  - integration
  - live provider
  - agent runtime
  - workflow runtime
  - coding runtime
- 测试必须区分：
  - 协议适配正确性
  - 运行时正确性
  - 平台联通正确性
- 不允许只依赖人工 demo 作为能力验证

## 18. 风险

- Provider 差异吸收复杂度高
- 统一输入输出与高级运行时之间存在边界冲突
- Workflow 与 Coding Agent 若范围失控，可能拖慢 2.0 主线
- 文档与测试若滞后，会导致 2.0 难以落地
