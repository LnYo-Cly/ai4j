# ai4j 2.0 实施计划

- 版本：v0.1-draft
- 日期：2026-03-19
- 状态：Draft
- 依赖文档：`docs/plans/2026-03-19-ai4j-2.0-constitution.md`

## 1. 实施目标

将 `ai4j 2.0` 从设计定稿推进到可编译、可测试、可文档化、可渐进迁移的实际工程状态。

## 2. 总体策略

- 先定边界，再迁代码
- 先收口运行时，再扩功能
- 先稳 `model + agent + workflow`，再深挖 `coding + cli`
- 兼容层持续保留，避免断崖式升级

## 3. 实施阶段

### Phase 1：模块骨架与文档定稿

目标：

- 创建 2.0 模块骨架
- 固化宪法文档
- 固化迁移原则

交付：

- 新 root `pom` 模块结构
- `docs/plans` 下正式设计文档
- 模块依赖关系初版

验收标准：

- Maven reactor 结构可解析
- 文档进入仓库
- 模块职责边界无明显冲突

### Phase 2：抽取 ai4j-core

目标：

- 建立稳定的基础契约层

交付：

- `ai4j-core` 包结构
- 通用异常、事件、trace、capability、tool SPI

验收标准：

- `ai4j-core` 不依赖上层模块
- 公共基础类型可复用

### Phase 3：重建 ai4j-model

目标：

- 统一模型服务层
- 保留简单体验

交付：

- `chat / response / embedding / image / audio / realtime / multimodal`
- facade 与兼容桥接

验收标准：

- 基础能力至少有 smoke demo
- 简单用户仍可快速接入

### Phase 4：统一 Loop Engine

目标：

- 收口 tool loop / agent loop 语义

交付：

- Loop Engine
- 生命周期事件
- 模型层自动 tool-loop 与 agent 运行时复用机制

验收标准：

- 不再新增 provider 自己控制完整 loop 的实现
- 运行时 trace 能覆盖工具调用链

### Phase 5：重建 ai4j-agent

目标：

- 建立通用 Agent Runtime
- 纳入 workflow

交付：

- `react / custom / subagent / team / workflow / trace / checkpoint`
- Workflow Canonical Schema

验收标准：

- workflow 运行时可单独验证
- agent 与 workflow 使用统一 runtime 语义

### Phase 6：建立 ai4j-coding

目标：

- 建立 coding agent 产品层

交付：

- coding session
- workspace
- file/shell/git/patch
- approval/sandbox/policy

验收标准：

- 能完成基本代码读写、命令执行、补丁应用流程
- trace 与 session 状态可观察

### Phase 7：兼容层、文档、测试

目标：

- 形成可发布状态

交付：

- 迁移指南
- 兼容清单
- 文档站新结构
- 分层测试矩阵

验收标准：

- 文档与 API 对齐
- 测试能区分协议问题与运行时问题

### Phase 8：CLI/TUI 首版

目标：

- 给 coding agent 提供首个本地入口

交付：

- CLI 命令结构
- 会话模式
- TUI 最小可用版

验收标准：

- 可运行本地会话
- 可连接 `ai4j-coding`
- 标记 `experimental` 或 `beta`
